## Phase 6: 지능형 공공기관/단체 필터링 (v2.6.8)
목표: 정부 기관(.go.kr), 공공/비영리 단체(.or.kr) 및 협회/학회 등 민감한 수신처를 자동으로 감지하여 발송을 건너뛰는 기능을 추가한다. 사용자가 원할 때만 작동하도록 토글(스위치) 기능을 제공한다.

- **Task 6-1 (필터 규칙 데이터 정의)**
  - 내장할 차단 도메인 패턴: `.go.kr`, `.or.kr`, `.re.kr`, `.ac.kr`, `.mil.kr`
  - 내장할 차단 키워드: `협회`, `학회`, `조합`, `중앙회`, `공사`, `공단`, `재단`
- **Task 6-2 (필터 엔진 `check_smart_filter` 구현)**
  - 입력받은 `email`이 차단 도메인으로 끝나는지 검사.
  - 입력받은 `comp_name`(업체명)에 차단 키워드가 포함되어 있는지 검사.
  - 하나라도 해당하면 `True`(차단 대상), 아니면 `False`를 반환하는 독립적인 함수 작성.
- **Task 6-3 (UI 토글 스위치 연동)**
  - [메시지 발송] 탭의 '자동발송 시작' 버튼 주변이나 적절한 위치에 **[☑️ 공공기관/단체 필터 적용 (go.kr, or.kr, 협회 등)]** 체크박스를 추가.
  - 이 체크박스의 상태 변수(`BooleanVar`)를 엔진 구동 시 파라미터로 넘길 수 있게 설정.
- **Task 6-4 (`real_engine` 안전한 결합 및 로깅)**
  - 발송 루프 안에서 수신처 1건을 꺼낸 직후, **블랙리스트 검사 전후**에 필터 적용 여부를 확인.
  - 체크박스가 ON이고 `check_smart_filter`가 `True`이면:
    - 로컬 DB(sent_log)나 구글 시트에 기록하지 않음.
    - 로그 출력: `🚫 필터링: {업체명} <{email}> (공공/단체 규칙 일치로 스킵됨)`
    - 즉시 `continue`를 호출하여 다음 수신처로 넘김. (v2.6.6의 메모리 최적화 루프 구조 파괴 금지)

## Phase 7: 계정 관리 UX 고도화 및 본문 기반 중복 차단 (v2.7.1)
목표: 다중 계정 사용자의 편의성을 위해 UI를 개선하고, 템플릿 이름이 아닌 '실제 메일 내용'을 기준으로 중복 발송을 차단하여 유연성을 극대화한다.

- **Task 7-1 (계정 리스트 UI/UX 전면 개편)** ✅
  - **이름 설정 & 삭제**: 계정 목록에서 특정 계정을 우클릭하거나 관리 버튼을 눌러 계정의 표시 이름(별칭)을 변경하고, 필요 없는 계정을 삭제할 수 있는 기능 추가 (`config.json` 동기화).
  - **계정 순서 변경**: 계정 목록에서 마우스 드래그 앤 드롭(Drag & Drop) 기능(또는 위/아래 이동 버튼)을 구현하여 사용자가 원하는 대로 계정 순서를 재배치할 수 있도록 구현.
  - **활성 계정 시각화**: 현재 사용자가 클릭해서 보고 있는(활성화된) 계정의 배경색이나 글씨체를 굵게 처리하여, 지금 어떤 계정을 작업 중인지 직관적으로 표시.

- **Task 7-2 (첨부파일 및 CID 초기화/편집 버그 수정)** ✅
  - **초기화 로직 강화**: 저장된 템플릿을 불러올 때, 기존에 화면에 남아있던 첨부파일 목록과 CID 목록을 **완벽하게 비운(Clear) 뒤** 템플릿의 데이터를 불러오도록 수정 (엉뚱한 파일 누적 추가 방지).
  - **개별 삭제 기능**: 로드된 첨부파일이나 CID 목록 옆에 [X] 버튼이나 삭제 기능을 명시적으로 제공하여, 템플릿 로드 후에도 특정 파일만 빼거나 새로 추가할 수 있도록 편집 기능 강화.

- **Task 7-3 (메일 본문 내용 기반(Hash) 중복 차단 - 로컬 DB 전용)** ✅
  - **문제점**: 기존에는 템플릿 이름(`template_name`)이 같으면 무조건 중복으로 처리됨.
  - **개선 로직**: 메일 내용의 변화를 감지하기 위해 발송 엔진(`real_engine`)에서 조립된 **메일 본문(HTML 내용)의 해시값(MD5 또는 SHA256)**을 추출.
  - **데이터베이스 반영**: 구글 시트 연동 로직은 속도 저하 방지를 위해 **절대 건드리지 않음**. 오직 로컬 SQLite DB의 `sent_log` 테이블 스키마에 `content_hash` 컬럼을 추가하여 기록함.
  - **중복 검사 기준 변경**: 중복 발송 판별 시(`check_duplicate_send_status`), 템플릿 이름이 아닌 **'동일한 `content_hash`를 가졌는지'**를 기준으로 스킵 여부를 결정. (내용을 한 글자라도 수정하면 새로운 메일로 인식)

- **Task 7-4 (전체 안정성 검증)** ✅
  - UI 이벤트 바인딩 시 메모리 누수나 기존 루프 충돌 방지.
  - JSON 파일(`config.json` 및 `templates.json` / `recipients.json` / `user_profiles.json`) 업데이트 시 데이터 유실 방지를 위한 안전한 파일 저장(원자적 저장) 구현.

## Phase 8: 블랙리스트(수신거부) 필터링 누수 긴급 패치 (v2.7.2)
목표: 수신차단 리스트에 등록된 이메일이 필터링을 통과하여 발송되는 치명적인 버그를 완벽하게 차단한다. 발송 엔진의 검증 순서와 데이터 대조(Matching) 로직을 전면 재점검한다.

- **Task 8-1 (블랙리스트 검증 함수 `check_blacklist` 강화)** ✅
  - 블랙리스트 DB와 현재 타겟 이메일을 대조할 때, 양쪽 모두 반드시 `.strip().lower()` 처리를 거치도록 강제하여 공백이나 대소문자 차이로 인한 필터링 누수를 원천 차단한다.
- **Task 8-2 (`real_engine` 검사 순서 및 분기 로직 정상화)** ✅
  - 현재 `real_engine` 루프 내에서 블랙리스트 검사 위치가 잘못되었거나 무시되고 있는지 확인한다.
  - **[반드시 지켜야 할 루프 순서]**: 수신처 1건 로드 ➡️ (1) 공공기관 필터 검사 ➡️ (2) **블랙리스트 검사** ➡️ (3) 본문 해시 중복 검사 ➡️ (4) MIME 조립 및 발송
  - 블랙리스트에 해당할 경우 반드시 로그에 `🚫 스킵: {업체명} (블랙리스트 차단)`을 출력하고, 즉시 `continue`를 호출하여 다음 수신처로 넘어가게(MIME 조립을 막게) 수정한다.
- **Task 8-3 (예외 상황 점검)** ✅
  - 블랙리스트에 등록된 값이 이메일이 아닌 도메인 형태(`@example.com`)일 경우에도 정상적으로 차단될 수 있도록 정규식(Regex) 또는 `endswith` 문자열 포함 검사를 보강한다.

## Phase 9: Amazon SES 및 특수 인증 방식 지원 (v2.7.3)
목표: Amazon SES와 같이 'SMTP 로그인 아이디'와 '실제 발신자 이메일 주소(From)'가 다른 환경을 지원하기 위해, 계정별 인증 방식을 선택(구 버전/신 버전)하고 데이터를 분리하여 처리한다.

- **Task 9-1 (인증 방식 선택 UI 추가)** ✅
  - [계정 설정] 창의 이메일 입력 영역 상단에 **인증 방식 선택 라디오 버튼**을 추가한다.
    - `[🔘 일반 SMTP (구 버전: 로그인 아이디=이메일)]` (기본값)
    - `[🔘 분리형 인증 (신 버전: Amazon SES 등)]`
  - '분리형 인증'을 선택했을 때만 UI에 **[보내는 사람 주소 (From)]** 텍스트 입력 칸이 부드럽게 나타나도록 동적 UI를 구성한다.
  - 기존의 입력 칸 라벨은 '분리형 인증' 선택 시 **[SMTP 로그인 아이디 (AKIA...)]** 로 변경하여 직관성을 높인다.

- **Task 9-2 (`config.json` 저장 로직 분리 및 호환성 유지)** ✅
  - 입력받은 데이터를 `config.json`의 해당 계정(`task_key`) 정보에 분리하여 저장한다.
  - 추가할 Key: `auth_type` ("standard" 또는 "separated"), `sender_email` (실제 수신자에게 보일 주소).
  - 기존에 등록된 계정들은 `auth_type`이 없더라도 자동으로 "standard"로 인식하도록 하위 호환성을 완벽히 보장한다.

- **Task 9-3 (`real_engine` 발송 봉투 조립 로직 수정)** ✅
  - SMTP 서버 로그인(`server.login`) 시에는 UI에서 입력받은 **[SMTP 로그인 아이디]**를 무조건 사용한다.
  - 메일 봉투 조립 시 `msg['From']`에 값을 넣을 때 `auth_type`을 검사한다.
    - "standard"일 경우: 기존처럼 로그인 아이디를 `From`으로 사용.
    - "separated"일 경우: 새로 입력받은 **[보내는 사람 주소 (From)]** 값을 사용.
